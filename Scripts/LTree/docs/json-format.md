# JSON format (`-j`)

`-j` always outputs the complete picture regardless of `-o` -- lines,
chars, totals, and the per-extension breakdown are all cheap to keep
around once computed, so JSON isn't filtered the way the tree view's
columns are. The point of `-j` is to hand the same data to something
else (a script, a dashboard, another tool) without scraping the
pretty-printed text.

## Top level

```json
{
  "path": "jsontest",
  "total": { "dirs": 2, "files": 2, "lines": 3, "chars": 12 },
  "by_extension": [
    { "ext": "txt", "files": 2, "lines": 3, "chars": 12 }
  ],
  "tree": { ... }
}
```

- `path` -- the scanned path, exactly as given on the command line.
- `total` -- same numbers as the `TOTAL` summary in tree mode.
- `by_extension` -- same table as the `FILES` summary, sorted by
  `lines` descending. Extensionless files are grouped under the
  literal string `"(no ext)"`.
- `tree` -- the recursive node structure, see below.

## Node object

```json
{
  "name": "x.txt",
  "type": "file",
  "symlink": false,
  "lines": 2,
  "chars": 8
}
```

Directories add two more fields:

```json
{
  "name": "a",
  "type": "dir",
  "symlink": false,
  "lines": 2,
  "chars": 8,
  "children": [ { ... }, { ... } ]
}
```

- `name` -- basename only (not a full path -- rebuild the path by
  walking `children` and accumulating names, same as you'd read the
  indentation in the tree view).
- `type` -- `"dir"` or `"file"`.
- `symlink` -- `true` if this entry is a symlink. Symlinked
  directories are never descended into (cycle safety), so a symlinked
  dir has `type: "dir"` but an empty or absent `children` array.
- `lines` / `chars` -- for a file, its own counts. For a directory,
  the sum over its **direct** children only (matches the `-o LINES`
  behaviour in tree mode, not a full recursive total -- sum up a
  subtree yourself if you need that).
- `truncated` -- present and `true` only on directories that exist but
  weren't expanded because of `-L`; such nodes have no `children` key
  at all rather than an empty array, so you can tell "no children"
  apart from "children not fetched".
- `children` -- only present on directories, in the same sorted order
  (case-insensitive, dirs and files interleaved) as the tree view.

## Full example

Generated by `ltree -j jsontest` against:

```
jsontest/
├── a/
│   └── x.txt   (2 lines)
└── b/
    └── y.txt   (1 line)
```

```json
{
  "path": "jsontest",
  "total": {
    "dirs": 2,
    "files": 2,
    "lines": 3,
    "chars": 12
  },
  "by_extension": [
    {"ext": "txt", "files": 2, "lines": 3, "chars": 12}
  ],
  "tree": 
  {
    "name": "jsontest",
    "type": "dir",
    "symlink": false,
    "lines": 3,
    "chars": 12,
    "children": [
      {
        "name": "a",
        "type": "dir",
        "symlink": false,
        "lines": 2,
        "chars": 8,
        "children": [
          {
            "name": "x.txt",
            "type": "file",
            "symlink": false,
            "lines": 2,
            "chars": 8
          }
        ]
      },
      {
        "name": "b",
        "type": "dir",
        "symlink": false,
        "lines": 1,
        "chars": 4,
        "children": [
          {
            "name": "y.txt",
            "type": "file",
            "symlink": false,
            "lines": 1,
            "chars": 4
          }
        ]
      }
    ]
  }
}
```
